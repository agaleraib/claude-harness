## 2. Capture `claude --version` in `.harness-state/` at session-start as a forensic breadcrumb

**Source:** [An update on recent Claude Code quality reports — Anthropic Engineering, 2026-04-23](https://www.anthropic.com/engineering/april-23-postmortem)

### Why it's relevant
The postmortem ties each regression to a specific build window (March 4 silent effort drop, March 26 session-clearing bug, April 16 verbosity instruction, all resolved by 2.1.116 on April 20). Maintainers who hit "felt off this week" had no way to correlate their experience with the build they were running, because Claude Code's version isn't surfaced anywhere in the harness's state files. The fix is a 3-line addition to `session-start`: capture `claude --version` (and the `.harness-profile`'s `model.primary` already-present pin) into `.harness-state/` once per session. If a future regression hits, `git log .harness-state/claude_version` reconstructs the timeline instantly.

This is *not* an ask for telemetry, alerting, or a dashboard — just a passive on-disk breadcrumb that makes a future "was it me, or did the tool change?" question answerable.

### Concrete change
Edit `skills/session-start/SKILL.md` Step 2 (the "Read current state" block at lines 28–43) to also write the active Claude Code version. Append after the existing parallel-read block:

```diff
 git status --short                                    # current working state
 ```

 The `git fetch` is intentional — it ensures branches pushed by automated routines (e.g. the Anthropic-reviews scheduler) are visible locally without moving master. Skip silently if there's no `origin` remote or no network.

+## Step 2b: Record a forensic breadcrumb (single line, no UI)
+
+```bash
+# Captures the Claude Code build at session open. If a future quality regression
+# hits, `git log .harness-state/claude_version` reconstructs which builds were
+# in use. Silent on missing CLI — never blocks session-start.
+command -v claude >/dev/null 2>&1 \
+  && claude --version 2>/dev/null > .harness-state/claude_version \
+  || true
+```
+
+The breadcrumb is intentionally write-only at session-start; nothing else reads it. Its value is forensic, not operational. Commit it (don't `.gitignore`) so timeline reconstruction works across machines — the file is one line, churn is low.
```

Tiny follow-up: add `claude_version` to the existing `.harness-state/` listing in `README.md` (if such a listing exists — verify before editing) so future readers know what it's for.

### Expected payoff
Zero cost in the steady state (one extra `command -v` + one file write per session). High payoff exactly once: the next time *something* feels off, `git log .harness-state/claude_version` answers "was I on a known-bad build?" without needing screenshots or memory.

### Why this isn't redundant with `.harness-state/last_exit.md` or `today_goal.md`
Both existing files record *user intent* (what was being attempted). `claude_version` records *tool state* (what Claude Code build was running). Different layers — last_exit answers "what was I doing"; claude_version answers "what was I doing it with."

**Verify before applying:**
1. `grep -n "claude --version\|claude_version" skills/session-start/SKILL.md` should return nothing — if it does, a version-capture step has already been added and this § is stale.
2. `ls .harness-state/claude_version 2>/dev/null` (in any project that has run `/session-start`) — if the file exists, the breadcrumb is already wired up somewhere.

**Recommended verdict:** apply — ~3 lines of bash, fails-closed (the `|| true` keeps session-start silent on machines without `claude` on PATH), pays off the next time a regression hits.
