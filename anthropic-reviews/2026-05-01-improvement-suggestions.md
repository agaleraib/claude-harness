## 1. Cite "Lessons from Building Claude Code: Prompt Caching is Everything" + add one-sentence `/clear`-vs-`/compact` nuance

**Source:** [Lessons from Building Claude Code: Prompt Caching is Everything — Thariq Shihipar, Anthropic / Claude blog, 2026-04-30](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything)

### Why it's relevant
The README's Session-Management section (`README.md:730–748`) opens with *"Context is your most precious resource. Everything else follows from this."* and then prescribes `/clear` between unrelated tasks as the *"single biggest quality improvement."* The prompt-caching post — written by an engineer on the Claude Code team — endorses the underlying premise (prompt prefix discipline is the architectural concern around which Claude Code itself was built) but adds one wrinkle the harness doesn't currently call out: **`/clear` resets the prompt cache, while `/compact` preserves the cached prefix up to the compact point.** For users on Claude Code Pro/Max this is invisible (Anthropic eats the cache miss). For users running Claude Code against the API directly, the same `/clear` advice that maximizes context quality also maximizes input-token cost.

The post quantifies this: *"Without prompt caching, a long Opus coding session can cost $50–100 in input tokens; with caching it costs $10–19."* That's not a small wedge.

The harness's existing recommendation isn't wrong — context quality usually trumps cost — but a one-sentence caveat lets API users make the trade-off explicitly.

### Concrete change
Two small edits to `README.md`:

1. **Append the post to the References list** (`README.md:987–996`):

   ```diff
   - [How We Built Our Multi-Agent Research System](https://www.anthropic.com/engineering/multi-agent-research-system) — Orchestrator-worker patterns, model routing
   - [Context Window Visualization](https://code.claude.com/docs/en/context-window) — Understanding what consumes your context
  +- [Lessons from Building Claude Code: Prompt Caching is Everything](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything) — Why static-prefix discipline (and `/compact` over `/clear`) matters when you're on metered API billing
   ```

   ```diff
  +
  +> **API-billing caveat:** `/clear` resets the prompt cache; `/compact` preserves the cached prefix up to the compact point. On Claude Code Pro/Max this is invisible. On the raw API, prefer `/compact` when context quality permits — see [Anthropic's prompt-caching post](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything) for the cost wedge (~5× input-token cost without cache hits on long Opus sessions).
   ```

### Why this isn't a methodology shift
The harness's One Rule (*"Context is your most precious resource"*) stays. The new line is a billing-mode footnote, not a doctrine change. Specifically: the harness still recommends `/clear` between unrelated tasks; it just acknowledges that the cost calculus differs by billing mode.

## 2. Add a "tools-as-cached-prefix" guardrail to `setup-harness` / hooks docs warning against dynamic MCP server swapping

**Source:** [Lessons from Building Claude Code: Prompt Caching is Everything — Thariq Shihipar, Anthropic / Claude blog, 2026-04-30](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything)

### Why it's relevant (scope: speculative)
The same post's third lesson — *"Lock tools at session start; changing the tool set mid-conversation is one of the most common ways people break prompt caching, because tools are part of the cached prefix"* — applies to any harness that dynamically adds or removes MCP servers during a session. The Claude Code internals use a `defer_loading: true` stub mechanism to keep the prefix stable.

The claude-harness today does not do anything dynamic with MCP servers. `setup-harness` installs skills/agents/procedures into the consumer project once; nothing in `skills/` or `.claude/agents/` toggles MCP servers at runtime. So the guardrail would be documentation for a problem we don't have.

None this run. Two paths when triaging:

1. **Easy path** — leave this as a defer. ## 3. Refactor orchestrator to reuse one subagent session across sequential tasks (cache amortization)

**Source:** [Lessons from Building Claude Code: Prompt Caching is Everything — Thariq Shihipar, Anthropic / Claude blog, 2026-04-30](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything)

### Why it's relevant (scope: speculative)
The post's framing — *"prompt caches are unique to models … if you're 100k tokens into a conversation with Opus and want to ask a question that is fairly easy to answer, it would actually be more expensive to switch to Haiku than to have Opus answer"* — surfaces a latent inefficiency in the orchestrator (`.claude/agents/orchestrator.md`): every dispatched subagent starts a fresh session, so the spec, the codebase context, and the routing reasoning are re-read from scratch on each task. A theoretical "coalesce sequential same-model tasks into a single subagent session" optimization would amortize the cache across 5–10 tasks instead of paying the prefix cost per task.

### Why this is a `reject` despite the technical opening
Three things stop this from being worth the redesign:

1. **Cost is Anthropic's, not the user's, on Pro/Max.** The harness's primary audience runs Claude Code on subscription billing. The cache wedge that motivates this in the post is real for Anthropic but invisible to the user.
2. **The orchestrator's per-task contract is the value prop.** Each subagent runs in isolation precisely so retries (Step 6 escalation rung), parallel dispatch (Step 5), and per-task commit (Step 7) all behave correctly. Coalescing sequential tasks into one session means a failure on Task 4 of 10 contaminates the context for Tasks 5–10.
3. **`isolation: "worktree"` for parallel tasks already requires fresh sessions.** The optimization would only apply to sequential same-model runs, which is a small fraction of orchestrator workload in practice.

The Anthropic post is written from the *Claude Code product team's* perspective (where every cache miss is a cost line). For a *consumer harness*, the dispatch contract matters more than the cache.

### Concrete change
None. Documenting the reject so a future "wait, can't we just batch these?" thought lands on this § and finds the trade-off already considered.

**Verify before applying:**
1. `grep -n "subagent\|spawn\|dispatch" /home/user/claude-harness/.claude/agents/orchestrator.md | head -20` should still show one-subagent-per-task semantics. If the orchestrator has been refactored to long-lived subagent sessions, this § is moot.
2. Check whether any consumer project has flipped `model_routing: off` and is using the orchestrator as a guided-spec executor — that mode already runs in the calling session (no subagent spawn) so cache reuse is automatic there. No change needed.

**Recommended verdict:** reject — premature optimization; cache cost is Anthropic's on subscription billing, and the per-task isolation contract is load-bearing for retries / parallel safety.
**Status:** PENDING — awaiting triage in PR review
