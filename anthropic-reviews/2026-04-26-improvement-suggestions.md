## 2. Add `procedures/triage-quality-issues.md` — "is it the model or your harness?" decision flow

**Source:** [A postmortem of three recent issues — Anthropic Engineering, 2026-04-23](https://www.anthropic.com/engineering/a-postmortem-of-three-recent-issues)

### Why it's relevant

The postmortem's headline operational finding: month-long reports of "Claude Code feels off" turned out to be three independent harness-layer changes — the **API was unaffected**. Users couldn't distinguish a model regression from a harness regression because the symptoms looked identical (forgetfulness, terse answers, missed reasoning). Anthropic explicitly says they're investing in "evaluations that can more reliably differentiate between working and broken implementations."

For a harness builder shipping skills/agents to consumer projects, the same triage gap exists locally. When `wordwideAI` (a downstream consumer per `procedures/`) reports "code-reviewer is missing things this week", the maintainer needs a fast way to ask: is this a model release issue, or did I change a SKILL.md or agent prompt recently? Today there's no convention for that — `procedures/` has security and migration checklists but nothing for quality-issue triage.

### Concrete change

New file: `procedures/triage-quality-issues.md` (~25 lines):

```markdown
# Triage: "the agent feels off"

When an agent (code-reviewer, ui-evaluator, generator, orchestrator) starts producing visibly worse output, work down this list **before** assuming the underlying model regressed. Anthropic's [April 23 postmortem](https://www.anthropic.com/engineering/a-postmortem-of-three-recent-issues) is a year-2026 reminder that month-long "model" complaints often resolve to harness-layer changes.

1. **What did I touch in the last 30 days?**
   `git log --since="30 days ago" --name-only -- .claude/agents/ skills/`
   If anything in that list is a prompt file (`*.md`), revert-test before anything else.

2. **What did Anthropic ship in the last 30 days?**
   Check `anthropic-reviews/reviewed-posts.md` for model-version changes since the symptom started. If a new Opus minor version landed, note its release date.

3. **Did the symptom start before or after the most recent prompt-file change?**
   If before any local change → likely upstream (Anthropic). If after → likely your harness. Bisect by reverting the most recent prompt edit on a branch and re-running the broken case.

4. **Try the API directly.**
   The April 23 postmortem isolated all three issues to the harness because the bare API was unaffected. Run the same task through `claude` CLI with a stripped system prompt — if quality returns, the problem is in your harness, not the model.

5. **Check `.harness-state/orchestrator.jsonl` for routing drift.**
   `effort` tier or model can silently change if `.harness-profile` was edited. `jq -r '[.task_id, .model, .effort, .ts] | @tsv' .harness-state/orchestrator.jsonl | tail -50` shows the last 50 dispatches.

If after all five steps the symptom persists with a clean harness and a stripped prompt, file an issue against Anthropic with a reproducer.
```

### Expected payoff

Closes a real gap: today there's no fast path from "agent feels off" → "is it me or them?" The procedure is ~30 lines, references infrastructure that already exists (orchestrator JSONL log from the 2026-04-19 spec, `.harness-profile` model pin, the anthropic-reviews tracker), and turns the postmortem's lesson into a checklist the maintainer can actually run on a Friday afternoon.

## 3. Re-evaluate the `stakes.level → effort_default` mapping now that Opus 4.7 product-defaults to `xhigh`

**Source:** [A postmortem of three recent issues — Anthropic Engineering, 2026-04-23](https://www.anthropic.com/engineering/a-postmortem-of-three-recent-issues)

### Why it's relevant

The postmortem's first issue: on March 4, Anthropic dropped Claude Code's default reasoning effort from `high` to `medium` and reverted on April 7 with this stronger default — *"4.6 family back to `high`; Opus 4.7 now defaults to `xhigh`."* The harness's current schema (per `docs/specs/2026-04-19-harness-model-pin-and-effort-routing.md` and the live `.harness-profile`) derives `effort_default` from `stakes.level`:

| stakes.level | effort_default |
|--------------|---------------|
| low          | medium        |
| medium       | high          |
| high         | xhigh         |

This repo's own profile is `stakes.level: medium → effort_default: high`. That's **one tier below** Anthropic's product default for Opus 4.7. A new project initialized with default stakes would also start one tier below the recommended product default.

The mapping is internally consistent (it's stakes-driven by design, per the spec's "Stakes-derived defaults" principle). But it was anchored before the April 23 postmortem clarified Anthropic's own intent. Worth a deliberate decision: leave the mapping (intentional divergence — harness is more frugal than Anthropic's product default) or re-anchor (track Anthropic's product default).

### Concrete change

This is a design pass, not a one-line fix. Spec out a follow-up to the 2026-04-19 model-pin spec that decides one of:

- **Option A — Re-anchor mapping.** Bump every tier up: low→high, medium→xhigh, high→xhigh. Edit the derivation comment in `.harness-profile`, the `project-init` schema docs (Wave 1 Task 2 of the 2026-04-19 spec), and the orchestrator's fallback rule (Step 4, rule 5).
- **Option B — Document divergence.** Keep the mapping as-is; add a sentence to the `model:` block comment in `.harness-profile` explicitly stating the harness runs one tier below Anthropic's product default for cost reasons, and citing the postmortem URL as the contemporaneous benchmark.
- **Option C — Per-task hint becomes the workaround.** Leave the mapping; document in README/orchestrator that high-stakes single tasks should use `**Effort:** xhigh` per-task hints (the mechanism already exists, see `orchestrator.md:60`).

Each option has a ~3-file blast radius (`.harness-profile`, `skills/project-init/SKILL.md`, `.claude/agents/orchestrator.md`) plus the original spec's Status note. None of the three is obvious without thinking through whether the harness should track Anthropic's product default or set its own (more conservative) default.

### Expected payoff

Resolves a real ambiguity introduced by the postmortem: today a reader of `.harness-profile` cannot tell whether `effort_default: high` is intentional frugality or stale-anchor drift. Either decision is fine; *no decision* leaves the project-init derivation rule one tier below the model vendor's recommended default with no comment about why.
