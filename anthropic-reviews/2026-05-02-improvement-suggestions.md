## 1. Consider an on-demand `/debug` skill that frames hypothesis → evidence → minimal repro → fix

**Source:** [Onboarding Claude Code like a new developer: Lessons from 17 years of development — Claude blog, ~2026-04-29](https://claude.com/blog/onboarding-claude-code-like-a-new-developer-lessons-from-17-years-of-development) (Brendan MacLean, MacCoss Lab — 700K-line C# codebase, 200K nightly tests, Claude Developer Ambassador case study)

**scope: speculative**

### What the post says
The MacCoss-Lab methodology has Claude treated like a new developer onboarding into a 17-year-old codebase: a thin `CLAUDE.md` at the root acts as the "lay of the land" (environment + pointers), and the project-specific *expertise* lives in skills. One of those skills, the *debugging skill*, is described as "designed to pull Claude out of 'guess and test' mode, pushing it toward root cause analysis before attempting any fix." The skill is invoked manually or attached to explicit conditions — no always-on cost.

### Why it could matter to claude-harness
The harness's "Safety Check Skills" tier already follows this exact shape — `/migration-check`, `/deploy-check`, `/a11y-check` are all on-demand, optional, and zero-context until invoked. A `/debug` (or `/root-cause`) skill would slot into the same tier:

```
/debug "the /translate endpoint returns 200 but body is empty when Authorization header is omitted"
→ skill prompts Claude to: (1) state hypotheses; (2) cite evidence (logs, file:line, test output);
   (3) write the minimal repro before any fix; (4) propose fix + confidence rating; (5) verify.
```

This is *not* a process-enforcement skill loaded at session start — it's an opt-in shape for one specific failure mode (debugging that drifts into "try the obvious thing, run it, change one variable, repeat").

### Why I'm recommending defer, not apply
1. `README.md:10` explicitly states *"Opus 4.7 plans carefully, sustains long tasks, **debugs effectively**, and follows instructions literally at low effort."* Shipping a debugging skill on the strength of an external case study would contradict the harness's own stated thesis without first-party evidence that "guess and test" is a real pain point in this maintainer's workflow.
2. The MacCoss case study runs against a 700K-line legacy C# codebase. Most consumer projects of this harness are smaller and younger; the bug-finding scaffolding that pays off in a 17-year-old codebase may be net cost in a 6-month-old one.
3. The harness's own anti-pattern rule (`anthropic-reviews/README.md:80–84`) is *"Don't add features beyond what the task requires."* Adding a debugging skill for a hypothetical pain point — when there's no `parking_lot.md` entry or postmortem in `.harness-state/` actually naming "guess and test" as a problem — is exactly the speculative bloat the routine is built to filter out.
4. The April-23 postmortem (already cited at `README.md:30`) names *silent-quality regressions in product Claude Code* as a real risk, but it doesn't argue for more in-harness debugging scaffolding — it argues for *measurable rubrics that catch regressions earlier*. That's the rubric direction (`criteria/`), not the process direction.

### What would flip this from defer to apply
A concrete maintainer-side incident, captured in `parking_lot.md` or a `.harness-state/` exit note, of the form *"spent 30+ minutes on a bug because Claude tried four fixes in a row without writing a repro first."* If two such incidents land within a quarter, the suggestion graduates: spec the skill via `/spec-planner`, then ship as `skills/debug/SKILL.md` with the same shape as `migration-check` (read-only framing, no auto-edits).

### Concrete change (for when it's promoted later)
Add `skills/debug/SKILL.md` with sections:

1. **Frame** — state the bug in one sentence; confirm scope (file path, function, observed vs expected).
2. **Hypotheses** — list 2-3 plausible causes; rank by likelihood; cite evidence per hypothesis.
3. **Minimal repro** — write the smallest test/script that reliably triggers the bug *before* attempting any fix. If a repro is impossible, say so explicitly and stop.
4. **Fix candidate** — propose one fix with a confidence rating (high / medium / low). For low-confidence fixes, propose a verification step.
5. **Verify** — run the repro; confirm fix or roll back.

Add one row to `README.md` under "Safety Check Skills":

```diff
 | `migration-check` | DB migration safety review | Before any schema change |
+| `debug` | Forces hypothesis → evidence → repro → fix when a bug resists the obvious fix | When a bug has burned ~15+ minutes of guess-and-test |
```

Then add the skill description to the user-level skill-copy loop in `README.md` lines 564–568. No agent-prompt changes; no profile-field changes; no orchestrator-routing changes.

### Expected payoff (if promoted)
Cuts the worst case of "Claude tries five fixes in a row, none stick" by an estimated 20-30 min/incident. Zero context cost when not invoked (same property as `/migration-check`).

### Easy example (for the maintainer to picture the shape)
```
You:    /debug "deploy-check claims env vars are set but the deploy fails with NoCredentialError"
Claude: Hypotheses:
        1. (likely) The check reads `process.env.AWS_ACCESS_KEY_ID` but the deploy reads from `~/.aws/credentials`
        2. (less likely) The env var is set in shell A, deploy runs in shell B
        3. (unlikely) IAM role override silently kicking in
        Evidence for #1: scripts/deploy-check.sh:42 reads env, scripts/deploy.sh:18 reads file. Different sources.
        Repro: `unset AWS_ACCESS_KEY_ID; bash scripts/deploy-check.sh && bash scripts/deploy.sh` — confirms.
        Fix candidate (high confidence): align deploy-check.sh to read from the same source as deploy.sh.
```

vs. without the skill, where Claude might just edit deploy.sh, fail, edit deploy-check.sh, fail, etc.

**Verify before applying:**
1. `ls skills/debug/ 2>/dev/null` — should return nothing today; if it returns `SKILL.md`, the skill already exists and this § is stale.
2. `grep -in "guess.and.test\|debug.*skill\|root.cause" parking_lot.md .harness-state/*.md 2>/dev/null` — if any maintainer-side incident matches, the "what would flip this to apply" condition is met and the verdict promotes from `defer` to `spec`.
3. Re-read `README.md:10` — if it has been edited to drop the "debugs effectively" claim (e.g. after a future Opus regression), the contradiction shrinks and the verdict is more comfortable as `spec` even without an in-house incident.

**Recommended verdict:** defer until a maintainer-side "guess and test" incident is captured in `parking_lot.md` or `.harness-state/` — pattern is real and concrete, but introducing process scaffolding ahead of in-house evidence violates the harness's own anti-pattern rule, and contradicts the stated thesis at `README.md:10`.
