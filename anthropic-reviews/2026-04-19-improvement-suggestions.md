## 2. Adopt the brain/hands split in the orchestrator agent

**Source:** [Scaling Managed Agents: Decoupling the brain from the hands — Anthropic Engineering, 2026-04-08](https://www.anthropic.com/engineering/managed-agents)

### Why it's relevant
The post's core lesson: keep Claude + its harness ("brain") on one side and tool execution ("hands") on the other side of a narrow `execute(name, input) → string` interface. This matches the gap between `orchestrator.md` (routing logic) and the Bash/MCP calls it actually dispatches — today they're implicitly coupled through the agent prompt.

### Concrete change
Refactor `agents/orchestrator.md` to describe explicit "hand" surfaces and make every tool call go through one of them:

- `hands/local_shell` → Bash in the repo
- `hands/playwright_mcp` → UI evaluator's browser
- `hands/github_mcp` → PR / issue IO

Then the routing table in the orchestrator can say *"for `run_tests`, dispatch to `hands/local_shell` regardless of which sub-agent is driving."* Today a reader has to reconstruct that mapping from scattered agent docs.

### Easy example
Before (implicit):
> "The ui-evaluator uses Playwright to test the app."

After (brain/hands split):
> "The ui-evaluator (brain) calls `hands/playwright_mcp.execute('navigate', {url})`. Swapping in a headless cloud browser later = replace one hand, zero changes to the agent."

### Expected payoff
Lets you host the same agent prompts under Claude Managed Agents later (their hosted sandbox *is* the hands) without rewriting anything. Also makes the "which tool does what" picture scannable in one file instead of five.

**Recommended verdict:** defer until Managed Agents adoption — abstraction pays off only when hosting on Claude Managed Agents; premature churn for solo tooling today.
**Status:** DEFERRED until Managed Agents adoption — premature abstraction for solo tooling · 2026-04-19

## 3. Restructure skills around progressive disclosure

**Source:** [Equipping agents for the real world with Agent Skills — Anthropic Engineering](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)

### Why it's relevant
Anthropic's official guidance: a skill is a folder with a `SKILL.md` (YAML frontmatter + instructions), and when `SKILL.md` gets unwieldy you split content into separate files referenced from it. This is the *exact* problem the README already flags ("context bloat: 22k+ tokens of skills loaded at startup"). The harness is philosophically aligned but not yet structurally aligned with the open standard.

### Concrete changes
1. **Rename `skills/<name>/` entry files to `SKILL.md`** (or add it as an alias). Right now `ls skills/session-start/` likely shows an ad-hoc filename — align with the open standard so these skills are portable to Claude.ai, the Agent SDK, and the Developer Platform without rewriting.

2. **Add YAML frontmatter to every skill.** Minimum fields: `name`, `description`, `when_to_use`, `triggers`. Example for `skills/commit/SKILL.md`:
   ```yaml
   ---
   name: commit
   description: Review → fix/park → commit → plan update cycle
   when_to_use: Invoke when the user says "commit", "ship it", or after code-reviewer passes
   triggers: ["/commit", "please commit", "let's ship"]
   ---
   ```

3. **Split the large skills.** `session-start/` probably bundles context injection + a daily kickoff prompt + instructions for the project-tracker. Separate those into referenced sub-files:
   ```
   skills/session-start/
     SKILL.md               # 20-line index + decision tree
     context-injection.md   # loaded only if no .harness-state/ exists
     daily-kickoff.md       # loaded only on first session of the day
   ```
   `SKILL.md` stays tiny; the 500-token budget claim in the README stops being a claim and becomes structurally enforced.

### Expected payoff
Aligns with Anthropic's open skill standard (cross-platform portability), enforces the lean-context promise mechanically, and gives you a free migration path for users who install via `setup-harness` on machines that already have Claude.ai or Agent SDK skills.

**Recommended verdict:** reject — factually wrong; all 11 harness skills already use `SKILL.md` with YAML frontmatter (PR's own Notes section flags this).
**Status:** REJECTED — change already in place across all 11 skills · 2026-04-19

## 4. Add a `claude-progress.txt` convention for long-running work

**Source:** [Effective harnesses for long-running agents — Anthropic Engineering](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)

### Why it's relevant
Anthropic's finding: the key artifact that lets a fresh-context agent pick up work mid-stream is a `claude-progress.txt` file alongside git history. The repo already has `parking_lot.md` (side-quests) and a project-tracker skill (session save/restore) but no in-progress state file for the current long task. These solve different problems — parking lot is *deferred* work, progress file is *current* work.

### Concrete change
Add a tiny convention documented in `procedures/`:

**New file: `procedures/claude-progress-convention.md`**
```markdown
# claude-progress.txt — one-paragraph convention

When an agent starts a task expected to span more than one session, it creates
or updates `claude-progress.txt` at repo root with:

- `Goal:` one sentence, the user's ask
- `Done:` bullets of finished sub-tasks, each ending with the commit SHA
- `Next:` the single next action
- `Blockers:` anything that needs the user

On session start, the session-start skill reads this file *before* anything else.
When the feature lands, delete the file in the same commit that closes the work.
```

Wire it in:
- Update `skills/session-start/SKILL.md` to read `claude-progress.txt` first if present.
- Update `skills/commit/SKILL.md` to ask "did this commit satisfy any `Next:`? update the file."
- Add a line to `.gitignore`? **No** — commit it, so the next session on another machine can resume.

### Easy example
Mid-way through a three-day refactor:
```
Goal: extract auth into its own package
Done:
  - moved middleware (sha a1b2c3)
  - moved token utils (sha d4e5f6)
Next: move the tests
Blockers: none
```
A fresh session running `/session-start` knows instantly what "next" is without re-reading the whole diff.

### Expected payoff
Closes the biggest gap Anthropic identifies in their own harness work with ~30 lines of documentation and two one-line skill edits.

**Recommended verdict:** reject — redundant with `.harness-state/last_exit.md` + `today_goal.md` + `parking_lot.md` + the `project-tracker` agent, which already cover cross-session handoff.
**Status:** REJECTED — redundant with existing `.harness-state/` + parking_lot + project-tracker mechanism · 2026-04-19

## 5. Formalize evals for the adversarial reviewers

**Source:** [Demystifying evals for AI agents — Anthropic Engineering](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)

### Why it's relevant
The harness's value prop is "adversarial evaluation agents that catch what self-assessment misses" (README line 13). But there's no way today to tell whether `code-reviewer.md` or `ui-evaluator.md` actually catch more than self-review — the agents themselves are unevaluated. Anthropic's post argues evals should be **verifiable end-state outcomes + rubrics**, which is *exactly* the shape this repo already has in `criteria/`.

### Concrete change
Create `criteria/agent-evals/` with one fixture per reviewer:

```
criteria/agent-evals/
  code-reviewer/
    01-missing-null-check.input/     # a repo snapshot with a known bug
    01-missing-null-check.expected   # "reviewer must flag line 42"
    02-passing-code.input/            # clean code
    02-passing-code.expected          # "reviewer must approve"
  ui-evaluator/
    01-broken-login-flow.input/
    01-broken-login-flow.expected     # "must fail UX rubric §3"
```

Add a `scripts/run-agent-evals.sh` that spawns each reviewer against each fixture and diffs against `.expected`. Run it before any change to a reviewer prompt. This is the *single-turn, verifiable end-state* pattern Anthropic describes.

### Easy example
Swap `code-reviewer.md` from "be thorough" to "be adversarial":
- Before eval: you guess it's better.
- After eval: fixture `01` previously passed with 2/5 detections, now passes with 5/5 → the change is justified.

### Expected payoff
Stops the rubrics from silently rotting. Turns "adversarial agents catch what self-assessment misses" from a claim into a measurement.

**Recommended verdict:** defer until consumer-project rubric suite exists — adversarial rubrics live in consumer projects' `criteria/`, not the harness; fixtures here would need a mock project.
**Status:** DEFERRED until a consumer-project rubric suite needs regression coverage — fixture scaffolding is high-effort for solo tooling today · 2026-04-19

## 6. Reflect the multi-agent supervision trend in the harness README

**Source:** [2026 Agentic Coding Trends Report — Anthropic](https://resources.anthropic.com/2026-agentic-coding-trends-report)

### Why it's relevant
Two data points from the report are directly about the kind of user this harness serves:
1. Developers use AI in ~60% of work but can "fully delegate" only 0–20% of tasks — i.e. supervision is the bottleneck.
2. Task horizons are expanding from minutes to days, and 2026 is about *coordinated teams of agents*, not single assistants.

The README's "Architecture" section sells a single-user flow ("you" running one agent at a time). The trend is multi-agent, parallel context windows, strategic human checkpoints.

### Concrete change
Add one short section to `README.md` — not a full rewrite — that names where this harness already supports parallel/supervised work and where it doesn't:

```markdown
## Multi-agent coordination (2026 trend)

Use these together for day-scale work:
- `orchestrator` → fans tasks out to sonnet/haiku in parallel
- `generator` + `code-reviewer` → the build/review loop, no human in between
- `ui-evaluator` → adversarial check, runs concurrently with code-reviewer
- `project-tracker` + `claude-progress.txt` → hand off between sessions

Human checkpoints (by design):
- After spec-planner (before coding starts)
- After code-reviewer (before commit)
- After ui-evaluator (before deploy)
```

### Easy example
Right now a new user reads the README and thinks "one agent, one task." After the edit they see the three-checkpoint loop and the parallel fan-out — matching how their team will actually want to use this in 2026.

### Expected payoff
Positions the harness against the current trend without adding new code — just honest naming of what's already there.

**Recommended verdict:** apply — cheap, honest naming of what already ships in `.claude/agents/` (orchestrator, generator, ui-evaluator, code-reviewer, project-tracker); pairs naturally with §1.
