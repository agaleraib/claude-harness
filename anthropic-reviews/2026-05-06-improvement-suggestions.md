## 1. Adopt prompt-caching guardrails ("static content first, dynamic content last") inside harness skills/agents

**Source:** [Lessons from building Claude Code: Prompt caching is everything — Claude Blog, 2026-04-30](https://claude.com/blog/lessons-from-building-claude-code-prompt-caching-is-everything)

### Why it surfaced
The post is a Claude Code team lessons-learned write-up. Headline takeaways:
- "Prompt caching is a prefix match. Any change anywhere in the prefix invalidates everything after it."
- "Static content first, dynamic content last."
- Don't put timestamps in the static system prompt (one cited example recovered "20+ percentage points of cache hit rate" by moving a timestamp to the bottom).
- Don't change tools mid-conversation; model state transitions (e.g. plan mode) as tool calls, not as system-prompt edits.
- Don't shuffle tool order definitions non-deterministically; don't update tool parameters mid-conversation.

The post is the closest in-window thing to a methodology insight from the Claude Code team. The natural question is: "do any of these guardrails translate into edits inside `skills/` or `.claude/agents/`?"

### Why I'm recommending reject
The harness's surface is markdown — `skills/<name>/SKILL.md` and `.claude/agents/<name>.md` files that Claude Code loads as agent and skill descriptions. It does not author or send Claude API requests. The post's lessons are about how an *application* (Claude Code itself) builds the request body that gets sent to the Anthropic API — request ordering, where timestamps live, whether to mutate the `tools` array between turns. Those are runtime decisions that Claude Code makes on the harness's behalf. The harness has no `cache_control` to set, no `system` array to reorder, no tools array to keep stable — Claude Code does that.

I considered four indirect angles and rejected each:

1. **"Heavy skill systems bloat the cached prefix → cite this post in `README.md` Why-Not-Superpowers."** Cache prefix length isn't a recurring cost — it's a one-time build, then cached. The README's existing argument is *context window consumption* (11% of context = 22k tokens), not cache-fill cost. Adding a "cache cost" bullet would dilute the existing strong arguments with a weaker one.

2. **"Add a guardrail forbidding skills from injecting timestamps."** Grep'd `skills/` and `.claude/agents/` for `date +\|$(date)\|new Date(`-style runtime timestamp injection into prompt text. Nothing matches — the only timestamps are inside markdown templates (`**Set at:** [timestamp]`) that are filled in by Claude when writing files, not by the harness when loading skills. So this would be defending against a non-existent problem (same shape as the deferred 2026-04-27 §2 verbosity-directive guardrail — *"Adding a guardrail for a problem that does not exist in the codebase is the same shape of bloat the harness is built to avoid"*). The existing 2026-04-28 §2 `claude_version` capture writes to a file (`.harness-state/claude_version`), not into a prompt — verified, not a cache concern.

3. **"Document that skills shouldn't change session-start behavior conditionally."** Same critique as (2). The session-start skill writes a `claude_version` breadcrumb to disk (added 2026-04-28 §2) but does not mutate Claude Code's tool list or system prompt. Verified.

4. **"Cite the post as an example of why measurable rubrics beat heuristic prompts."** The post is about API engineering, not about evals or rubrics. Citing it as evidence for the harness's lean-context thesis would be a stretch.

If the harness ever grew a skill or agent that *did* construct Claude API requests directly (e.g. a future `/scan-codebase` skill that called the Claude API in a loop, bypassing Claude Code), this post would become directly actionable. That's a future problem; flag it then.

### Adjacent: there's already a `claude-api` skill upstream
The post's exact lessons are already encoded in Anthropic's user-level `claude-api` skill (visible in the available-skills list of any Claude Code session: *"Apps built with this skill should include prompt caching."*). That skill is **not** in this repo — it ships from Anthropic to all Claude Code users. So projects that install this harness via `setup-harness` and then use `/claude-api` to scaffold Anthropic SDK code will inherit those guardrails for free, without this repo needing to mirror them.

### Expected payoff (if applied — for completeness)
Zero, by my reading. The recommendations are runtime API-client patterns; the harness has no API client. A defensive guardrail line in `anthropic-reviews/README.md` would cost a paragraph and protect against a problem that the codebase doesn't currently exhibit and has no clear path to exhibiting.

**Verify before applying:**
1. `grep -rinE 'cache_control|prompt[_ -]?cach|@anthropic-ai/sdk|anthropic\.messages\.create' skills/ .claude/agents/ examples/ AGENTS.md README.md` — if any match appears, the harness has gained an API-client surface and the post becomes directly relevant; promote this § from `reject` to `apply`.
2. `grep -rinE 'date \+|new Date\(|datetime\.now' skills/ .claude/agents/` — if a skill is found injecting a runtime timestamp into prompt text, the angle (2) defense becomes worth implementing.
3. Re-check whether `setup-harness` has started bundling a fork of the user-level `claude-api` skill (it currently doesn't, per `grep -n "claude-api" skills/setup-harness/SKILL.md` returning nothing); if so, those bundled lessons may need a refresh to match this post.

**Recommended verdict:** reject — the harness has no Claude API client surface; the post's lessons are runtime request-construction patterns. Captured here so a future reader sees the analysis was done and four indirect angles were considered.
