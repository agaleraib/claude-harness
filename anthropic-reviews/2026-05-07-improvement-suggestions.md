## 3. PostToolUse `updatedToolOutput` now works for all tools — context-compression hook pattern

**Source:** Claude Code release notes, May 2026 ([changelog](https://code.claude.com/docs/en/changelog)) — *"PostToolUse hooks can now replace tool output for all tools via `hookSpecificOutput.updatedToolOutput` (previously MCP-only)."* Plus *"PostToolUse and PostToolUseFailure hook inputs now include `duration_ms`."* See also the GitHub feature request: [anthropics/claude-code#32105](https://github.com/anthropics/claude-code/issues/32105).

### Why it's relevant
Tool results consume ~60% of context tokens in agentic Claude Code sessions per Anthropic's own measurements. The new capability lets a hook compress, filter, or summarize verbose tool output (Bash with long stderr, MCP responses, etc.) before it lands in the agent's context — for *any* tool, not just MCP. This is a real new lever.

For this repo specifically: `/home/user/claude-harness/.claude/settings.json` currently has `"hooks": {}` (empty) — intentionally — and the `README.md` already mentions PostToolUse as a *concept* without wiring it up. So this isn't an "already done" case; it's a new affordance the harness has chosen not to use yet.

### Why I'm recommending defer, not apply
1. **No measured pain.** The harness is solo-maintained on a tooling repo with short, focused sessions. There's no evidence of context-budget pressure today.
2. **Speculative hooks fight the "Why Not Superpowers" thesis.** Adding a generic context-compression hook would be exactly the kind of harness-layer instruction that the April 23 postmortem flagged as quality-degrading when added speculatively.
3. **The right time is when a consumer project hits the budget ceiling.** Then the design (which tools to compress? truncate vs. summarize? preserve what?) is grounded in observed behavior.

### What to capture now
- Add a one-line item to `parking_lot.md` under "Open" pointing at this § so it surfaces when token-budget pain shows up:
  ```
  - PostToolUse `updatedToolOutput` is now available for all tools (not just MCP) — wire up a context-compression hook IF a consumer project starts hitting tool-output token bloat. See anthropic-reviews/2026-05-07-improvement-suggestions.md §3.
  ```
- No `.claude/settings.json` change today; the empty hooks block is the right answer until pain is measured.

### Expected payoff (when eventually applied)
Tool-result compression can recover meaningful context budget on long sessions per Anthropic's own audits. Worth capturing as a tool in the kit, not worth pre-wiring.

**Verify before applying:** `cat .claude/settings.json` — if `hooks` is no longer `{}`, the maintainer has started using hooks for something else and the design space is no longer empty; rethink the suggestion. Also check `parking_lot.md` for an existing token-budget item before adding a duplicate.

**Recommended verdict:** defer until token-budget pain is real — capability is genuinely new, but premature to wire up; park in `parking_lot.md` for traceability.
