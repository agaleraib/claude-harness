## 1. Adopt MacLean's "CLAUDE.md = lay of the land, skills = expertise" framing in the README's CLAUDE.md guidance

**Source:** [Onboarding Claude Code like a new developer: Lessons from 17 years of development — claude.com/blog, 2026-04-28](https://claude.com/blog/onboarding-claude-code-like-a-new-developer-lessons-from-17-years-of-development)

### Why it's relevant
MacLean (Claude Developer Ambassador, MacCoss Lab) describes onboarding Claude to a 700K-line C# codebase by **separating two layers**: a `CLAUDE.md` at the root that handles environment + points Claude to docs ("the 'lay of the land,' not the expertise itself"), and *skills* that carry the project-specific expertise (his examples: a `skyline-development` orientation skill, a `version-control` skill encoding commit/PR conventions, a `debugging` skill that pulls Claude out of guess-and-test mode).

The harness's README already enforces *quantitative* CLAUDE.md hygiene rules (`README.md:646-651` — "Under 200 lines. Under 50 is better. Only include what Claude would get wrong without the instruction. If Claude already does it correctly, delete the rule. Treat it like code: prune regularly.") but does *not* state the *qualitative* layer separation. MacLean's framing — CLAUDE.md is orientation, skills are expertise — is a sharper articulation of the same lean discipline and now has a first-party blog-post citation. Worth grafting into the existing rules block as a one-paragraph clarifier.

### Concrete change
Append a sentence to the rules block at `README.md:646-651`:

```diff
 **Rules for CLAUDE.md:**
 - Under 200 lines. Under 50 is better.
 - Only include what Claude would get wrong without the instruction
 - If Claude already does it correctly, delete the rule
 - Treat it like code: prune regularly
+- **Layer separation**: CLAUDE.md is the *lay of the land* (build commands, where docs live, non-obvious project conventions). Project-specific *expertise* belongs in skills, not here. If a rule starts to read like a tutorial, it's a skill — extract it. ([source](https://claude.com/blog/onboarding-claude-code-like-a-new-developer-lessons-from-17-years-of-development))
```

Optional second touch: in the "Why Not Superpowers / Heavy Skill Systems?" section (`README.md:23-32`), the existing bullets explain *why* skills shouldn't bloat CLAUDE.md but don't give a positive prescription for *where* expertise should go. This new rule closes that gap by name.

### Expected payoff
A consumer-project maintainer reading the harness's CLAUDE.md guidance gets the *positive* version of the rule (where to put expertise) alongside the *negative* one (don't bloat CLAUDE.md). Reduces the ambiguity that produces 500-line CLAUDE.mds in the first place — the failure mode the README explicitly calls out at line 847.

### Why it's not redundant with existing bullets
The existing rules describe *what to remove* from CLAUDE.md. They do not describe *what to do with* removed expertise — readers default to losing it or stuffing it back later. The MacLean framing names skills as the destination, which closes the loop.

**Verify before applying:**
1. `grep -n "lay of the land" README.md` should return nothing today; if it does, the framing has already been added and this § is stale.
2. `grep -nA8 "Rules for CLAUDE.md" README.md` should show the existing four bullets without a layer-separation entry.

**Recommended verdict:** apply — three-line README edit, sharpens an existing rule with first-party language, no design risk.

## 2. Promote `docs/prompts.md` "Trace a bug" prompt to a `/debug` skill that codifies root-cause-before-fix

**Source:** [Onboarding Claude Code like a new developer: Lessons from 17 years of development — claude.com/blog, 2026-04-28](https://claude.com/blog/onboarding-claude-code-like-a-new-developer-lessons-from-17-years-of-development)

### Why it's relevant
MacLean's most-cited skill in the post is a *debugging* skill explicitly designed to "pull Claude out of guess-and-test mode, pushing it toward root-cause analysis before attempting any fix." The harness already has the *content* of this skill — `docs/prompts.md:202-213` is a "Trace a bug from symptom to cause" prompt that ends with "Don't fix anything yet — just find the root cause." The difference is the activation surface: a prompt in a docs file is a copy-paste affordance; a skill is invokable as `/debug` with progressive-disclosure load semantics.

If the harness wants the same anti-guess-and-test discipline MacLean reports, the path is to formalize the existing prompt as a skill rather than leave it in `prompts.md` where it competes with 30+ other ad-hoc snippets for the user's attention.

### Concrete change (if approved)
1. Create `skills/debug/SKILL.md` with the existing `docs/prompts.md:202-213` prompt as the body, plus a one-paragraph rule block ("Do not edit any file in this skill before stating the root cause hypothesis.") and a verification step ("After identifying the cause, before fixing: state the hypothesis in one sentence and confirm with the user.").
2. Remove the duplicate prompt from `docs/prompts.md` and replace it with a one-line pointer (`See /debug skill.`).
3. Add `/debug` to the README's skill catalog under a new "Diagnostic" section.

**~3 file edits, ~40 lines net added.** Borderline-trivial in size; the *justification* is what's substantive.

### Why this is `defer`, not `apply`
1. **Lean-over-heavy is the harness's marquee thesis.** Adding a skill — even a small one — is a context-surface decision the README defends specifically against (`README.md:23-33`). Adding skills "because Anthropic blogged about one" cuts against the discipline.
2. **The prompt already exists and is reachable.** A user looking for a bug-trace prompt finds it at `docs/prompts.md:202`. No reported friction with that location.
3. **MacLean's context is enormous (700K LoC, 17 years).** Root-cause-before-fix matters most when the cost of a wrong-fix is days of regression debugging. The harness's median consumer-project is much smaller; the existing prompt's ROI is unproven at smaller scale.
4. **Similar role exists.** `code-reviewer` already catches "did this fix the symptom or the cause?" issues post-hoc in its rubric.

**Promotion path** — if a consumer project (e.g. wordwideAI, finflow) reports that bugs get patched without root-cause analysis *and* the existing `docs/prompts.md` entry isn't being invoked, that's the trigger to promote this § from `DEFERRED` to `SPEC'd` and run `/spec-planner` on it. Until then it sits.

scope: speculative

**Verify before applying:**
1. `ls skills/debug/SKILL.md 2>/dev/null` should return nothing today; if it returns a file, the skill exists and this § is moot.
2. `grep -n "Trace a bug" docs/prompts.md` should still find the prompt — if it's already been moved or removed, the basis for this § has changed and it should be re-evaluated.
3. `grep -n "guess.and.test\|guess-and-test" skills/` should return nothing — confirms the framing isn't in any existing skill.

**Recommended verdict:** defer until a consumer project surfaces a "fixes-symptom-not-cause" failure that the existing `docs/prompts.md` prompt didn't prevent. The lean-over-heavy thesis carries the burden of proof, and right now there's no failure case demanding more skill surface.
