## 1. Note the v2.1.126 `--dangerously-skip-permissions` semantic widening in `triage-quality-issues.md` as a tool-version behavior change

**Source:** Claude Code [CHANGELOG](https://github.com/anthropics/claude-code/blob/main/CHANGELOG.md), entry for **v2.1.126** released **2026-05-01** (cross-referenced via [releasebot summary for 2026-05-01](https://releasebot.io/updates/anthropic/claude-code))

### Why it's relevant
v2.1.126 carries a small but real semantic shift to a flag that exists specifically for automation:

> `--dangerously-skip-permissions` now bypasses prompts for writes to `.claude/`, `.git/`, `.vscode/`, shell config files, and other previously-protected paths.

Before 2.1.126 the flag bypassed *most* permission prompts but still surfaced a confirmation for writes to those protected paths. After 2.1.126 it does not. The blast radius of an automated `claude --dangerously-skip-permissions ...` invocation is therefore wider on 2.1.126+ than on 2.1.122 or earlier — silently — and the only on-disk evidence the user has is the `claude --version` breadcrumb that 2026-04-28 §2 added (`skills/session-start/SKILL.md:51-53`). The triage procedure today (`procedures/triage-quality-issues.md`) walks the user through "is it the model or your harness?" but doesn't currently say "or is it the *tool* — Claude Code itself — having shifted behavior between versions?". The 2.1.126 change is a clean illustrative example to introduce that third axis.

This is preventive, not corrective: the harness's own scripts and skills do **not** invoke `claude --dangerously-skip-permissions` anywhere (verified by `grep -rn "dangerously-skip-permissions" /home/user/claude-harness/` returning zero matches). The risk surfaces only when a *consumer project* (a CI hook, a cron job, an automation script) uses the flag — at which point the same script behaves differently across the 2.1.122 → 2.1.126 boundary with no warning.

### Concrete change
Append a sixth step to `procedures/triage-quality-issues.md`:

```diff
 5. **Check `.harness-state/orchestrator.jsonl` for routing drift.**
    `effort` tier or model can silently change if `.harness-profile` was edited. `jq -r '[.task_id, .model, .effort, .ts] | @tsv' .harness-state/orchestrator.jsonl | tail -50` shows the last 50 dispatches.

+6. **Check whether the Claude Code CLI itself shifted behavior between versions.**
+   `git log -p .harness-state/claude_version` (the breadcrumb captured at session-start) shows when the build changed. Cross-reference any version transition near the symptom date against the [Claude Code CHANGELOG](https://github.com/anthropics/claude-code/blob/main/CHANGELOG.md) for behavior changes — not all of them are model regressions. Concrete example: v2.1.126 (2026-05-01) widened `--dangerously-skip-permissions` to bypass writes to `.claude/`, `.git/`, `.vscode/`, and shell config files; an automation script that crossed that boundary will silently have a wider blast radius after the upgrade than before.

 If after all five steps the symptom persists with a clean harness and a stripped prompt, file an issue against Anthropic with a reproducer.
```

(Update the trailing line from "all five steps" to "all six steps" while editing.)

### Expected payoff
Closes the loop between the existing forensic breadcrumb (added 2026-04-28 §2) and the existing triage flow. Today the breadcrumb has no documented consumer — it was written for "future use." This is the future use. Without this edit, the next person who hits a tool-behavior shift will still re-discover the linkage by hand.

### Why it's recommended `defer`, not `apply`
The harness itself never uses `--dangerously-skip-permissions`, so there is no live regression scenario this edit prevents *today*. The value materializes the first time a consumer project's automation crosses a Claude-Code-version boundary that changes flag semantics. Until that happens, the procedure addition is speculative documentation. The trigger to promote from `defer` to `apply` is concrete: the first time a consumer project (wordwideAI, gobot, or any future addition) introduces a script that runs `claude --dangerously-skip-permissions`, this § becomes worth shipping immediately.

**Verify before applying:**
1. `grep -n "dangerously-skip-permissions" procedures/triage-quality-issues.md` should return nothing — if it does, the note has already been added and this § is stale.
2. `grep -rn "dangerously-skip-permissions" /home/user/claude-harness/ skills/ scripts/` should still return zero — if any script has *started* using the flag, promote this § from `defer` to `apply` immediately.
3. `grep -n "claude_version" skills/session-start/SKILL.md` should still return the breadcrumb capture (the diff above assumes it exists; if it has been removed, fix that first).

**Recommended verdict:** defer until consumer-project automation uses `--dangerously-skip-permissions` — preventive documentation; the harness itself doesn't trigger the new behavior, but the breadcrumb infrastructure is already in place to make this triage step real the moment it does.

## 2. Cross-reference `claude project purge [path]` from `setup-harness` as a clean-reset path

**Source:** Claude Code [CHANGELOG](https://github.com/anthropics/claude-code/blob/main/CHANGELOG.md), entry for **v2.1.126** released **2026-05-01**:

> Added `claude project purge [path]` to delete all Claude Code state for a project (transcripts, tasks, file history, config entry).

### Why it was considered
A new top-level CLI verb that "deletes all Claude Code state for a project" sounds adjacent to `setup-harness`'s install path — a symmetric uninstall surface, in principle.

### Why it's recommended `reject`
Two reasons, either of which alone is sufficient:

1. **`setup-harness` has no uninstall path.** Inspecting `skills/setup-harness/SKILL.md` shows it copies skills, agents, and criteria into `~/.claude/` and project-local directories, but never advertises a teardown command. Adding documentation for `project purge` would be the *first* uninstall touchpoint the skill mentions — which is a documentation expansion, not a cross-reference. If teardown becomes a real need, it deserves its own dedicated § with a designed UX, not a sidebar in the install skill.

2. **The state surfaces are disjoint.** `claude project purge` deletes Claude Code's per-project state: chat transcripts, tasks, file history, and the project's entry in `~/.claude.json`. The harness's durable state lives in `.harness-state/`, `docs/plan.md`, `docs/specs/`, `docs/waves/`, and `parking_lot.md` — none of which `project purge` touches (and rightly so; harness state is checked into git, Claude Code's per-project state is local-only). Documenting `project purge` inside `setup-harness` would imply a teardown relationship that doesn't actually exist and would mislead future readers about what gets removed.

### Concrete change
None. This § exists in the durable record so a future review doesn't re-surface the same suggestion when v2.1.126 falls outside the search window.

**Verify before applying:** Not applicable — recommended verdict is `reject`. If the maintainer wants to revisit (e.g. `setup-harness` grows an uninstall path), the entry to read is the §1 of the eventual install-uninstall design, not this one.

**Recommended verdict:** reject — premature (no uninstall path exists in `setup-harness`) and conceptually misleading (Claude Code project state ≠ harness durable state).
**Status:** PENDING — awaiting triage in PR review
