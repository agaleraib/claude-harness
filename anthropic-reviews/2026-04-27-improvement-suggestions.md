## 1. Align `effort_default` derivation with Anthropic's new Opus 4.7 default of `xhigh`

**Source:** [An update on recent Claude Code quality reports — Anthropic Engineering, 2026-04-23](https://www.anthropic.com/engineering/april-23-postmortem)

### Why it's relevant
The postmortem closes with: *"all users now default to xhigh effort for Opus 4.7 and high effort for all other models."* Anthropic's own product (Claude Code) has shifted its baseline upward — and the central root cause traced in the postmortem was that lowering effort silently degraded coding quality enough to look like a model regression for an entire month.

The harness's `effort_default` derivation in `skills/project-init/SKILL.md` (lines 138–144) and the inline comment template (lines 176–181) currently produces:

| `stakes.level` | derived `effort_default` |
|---|---|
| `low`    | `medium` |
| `medium` | `high`   |
| `high`   | `xhigh`  |

Since `model.primary` is `claude-opus-4-7`, the medium-stakes default of `high` is now strictly below Anthropic's published default for the same model — and the harness's own `.harness-profile` is exhibit A (line 31: `effort_default: high   # derived from stakes.level: medium`).

### Concrete changes
- **`skills/project-init/SKILL.md` lines 142–144** (the derivation table): bump every row up one rung except `xhigh` (the cap):

  ```diff
  - | `low`    | `medium` |
  - | `medium` | `high`   |
  - | `high`   | `xhigh`  |
  + | `low`    | `high`   |
  + | `medium` | `xhigh`  |
  + | `high`   | `xhigh`  |
  ```

  Add one sentence under the table noting the alignment: *"As of Anthropic's 2026-04-23 postmortem, Claude Code defaults Opus-4.7 users to `xhigh`. The derivation above stays at or above that floor for the primary model."*

- **`skills/project-init/SKILL.md` lines 177–180** (inline comment template emitted into every new `.harness-profile`): mirror the same numbers so freshly-generated profiles carry the new derivation in their own header comment.

- **`.harness-profile` line 31** (this repo's own profile): bump `effort_default: high` → `effort_default: xhigh`. The repo carries `stakes.level: medium`, so under the new derivation it should be `xhigh`. Update the trailing comment to match: `# derived from stakes.level: medium`.

- **No changes needed** in `agents/orchestrator.md` (lines 154–166): rule 5 already reads `effort_default` from `.harness-profile` rather than re-deriving from `stakes.level`, so the rule stays correct after the table changes.

- **No retroactive migration**: the derivation is "first-write only" by spec (line 146), so existing consumer projects keep whatever they hand-tuned. Only freshly-init'd projects get the new defaults. This matches the existing override semantic.

### Easy example
Before, a `stakes.level: medium` repo (the most common bucket) inherited `effort_default: high`. After the postmortem confirmed Anthropic has shifted its own product's floor, the same repo inherits `effort_default: xhigh` — which the orchestrator's rule 5 then propagates to the long, multi-step subagent dispatches that benefit most from the extra reasoning.

### Expected payoff
Brings the harness's tuning back into agreement with Anthropic's tuning for the primary model. Removes a silent gap (the harness was below Anthropic's own default after 2026-04-23) without forcing migrations on existing projects. The cost is straightforward: more tokens per dispatch on medium-stakes repos. Users who care about cost can override `effort_default: high` directly — the override path is already documented and doesn't change.

**Verify before applying:** re-read `skills/project-init/SKILL.md` lines 138–146 and the corresponding comment block at lines 177–181; confirm Anthropic's postmortem text at https://www.anthropic.com/engineering/april-23-postmortem still names `xhigh` as the Opus-4.7 default; check that `.harness-profile` line 31 still reads `effort_default: high`.

**Recommended verdict:** apply — small mechanical edit in two places + this repo's own profile; brings harness in line with Anthropic's published default for the primary model; existing consumer projects unaffected because derivation is first-write-only.

## 2. Add a "no verbosity-reduction directives in skill/agent prompts" guardrail

**Source:** [An update on recent Claude Code quality reports — Anthropic Engineering, 2026-04-23](https://www.anthropic.com/engineering/april-23-postmortem)

### Why it's relevant
The postmortem identifies a single-line system-prompt verbosity instruction (added 2026-04-16, reverted 2026-04-20) as one of the three changes that hurt Claude Code's coding quality for the month leading up to the report. The lesson is: prompt-engineering instructions that look like "trim the output" can have outsized effects on code quality, because the model uses output tokens for reasoning and intermediate scaffolding even when the user-visible answer is brief.

The harness ships ~14 skill prompts and 6 agent prompts that target Claude Code. A future maintainer (you in 6 months, or a contributor) could plausibly add a "be more concise" line to a skill in good faith — and the failure mode (subtle code-quality regression in consumer projects, no visible error) is hard to attribute back to the prompt change.

### Verification — current state of prompts
Grep'd `skills/` and `.claude/agents/` for likely directives — `be concise`, `keep.*concise`, `short response`, `less verbose`, `reduce verbos`, `minimize output`. **Zero matches.** The handful of `brief|short|concise` hits are unrelated (`git status --short`, "brief the subagent", "Add a brief note with the commit hash").

So the guardrail is not currently violated. The suggestion is a *defensive* convention to keep it that way.

### Concrete changes (if the maintainer chooses to apply)
A single line under the existing `## Anti-patterns to avoid` section in `anthropic-reviews/README.md` (or a new `procedures/prompt-engineering-guardrails.md`) of the form:

```markdown
- **Don't add verbosity-reduction directives to skill/agent prompts.** Per Anthropic's
  2026-04-23 Claude Code postmortem, a single "reduce verbosity" instruction was one
  of three changes that silently hurt coding quality for a month. The harness
  ships prompts that target Claude Code; the same risk applies. If a skill output
  feels too long, change the structure, not the instruction.
```

### Why I'm recommending defer instead of apply
The harness's own README explicitly warns against "Don't add features, refactor, or introduce abstractions beyond what the task requires." The grep confirms zero offending lines today. Adding a guardrail for a problem that does not exist in the codebase is the same shape of bloat the harness is built to avoid.

The cleaner shape is to *keep this suggestion in the dated file as a watch-item* — so when a future PR adds a "be concise" directive, the reviewer can search the anthropic-reviews/ history, find this §, and turn it down with a one-line citation.

### Expected payoff (if applied later)
Prevents a silent code-quality regression in any project that installs the harness via `setup-harness`. Cost is one paragraph of documentation. Most valuable when the harness grows past its current ~20 prompts.

**Verify before applying:** before promoting this from `defer` to `apply`, re-run `grep -rinE 'be concise|keep.*concise|short.*response|less verbose|reduce verbos|minimize output' skills/ .claude/agents/`. If the grep is still empty, the guardrail is still purely defensive — `defer` remains the right call. If the grep returns matches, promote to `apply` and use the matched files as the motivating example in the guardrail text.

**Recommended verdict:** defer until a verbosity directive actually appears in a skill or agent prompt — current grep is empty, so adding the documentation today is speculative bloat by the harness's own anti-pattern rules.
